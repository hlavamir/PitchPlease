# Version 2 — WS2811 Addressable Strips

> Switch to addressable WS2811 LED strips, allowing per-segment color control. Three sub-versions: v2.0 (2 strips), v2.1 (4 strips, Arduino R3, limited bandwidth), v2.2 (4 strips, Arduino R4, full independent control).

## Overview

V2 introduced WS2811 12V addressable LED strips, where every ~5 cm segment can be set to a different color independently. This unlocked proper audio-reactive visuals driven by the vvvv patch. Both sub-versions communicate via serial and use FastLED. Patterns are always symmetric — only half the LED data is transmitted per strip, and the firmware mirrors it.

## Physical Construction

Each firmware "strip" (one data channel) is housed inside a **1m triangular aluminium profile**. Inside each tube are two physical WS2811 strips:

- **Strip A:** 19 segments, running left to right along one face of the triangle
- **Strip B:** 19 segments, running right to left along the adjacent face — oriented in the opposite direction, with its start soldered to the end of Strip A via 3 short wires

The two strips are placed at a slight offset so their LEDs interleave, filling each other's gaps for a denser appearance. Because Strip B runs in reverse, addressing LEDs 0–18 and 19–37 with the same values (just mirrored) produces a visually symmetrical signal from the viewer's perspective. This is why the firmware only receives 19 unique color positions per channel and mirrors them — the physical reversal of Strip B handles the rest.

**Total:** 38 LEDs per channel (19 + 19), in a 1m profile.

## V2.0 — 2 Strips

**Firmware:** `v2_ws2811/firmware/pitch_please_w2811/pitch_please_w2811.ino`

**Serial baud rate:** 57600

| Parameter | Value |
|---|---|
| Strips | 2 |
| LEDs per strip | 38 (physical) |
| Unique positions received | 10 (mirrored by firmware) |
| Data pins | D7 (strip 0), D8 (strip 1) |
| LED type | WS2811, FastLED |
| PSU | 12V, 3A |

**Protocol:** vvvv sends `(2 strips × 10 positions × 3 bytes RGB) + 2` bytes per frame. The `+2` are a mode-switch byte and a confirmation byte (255).

10 values per strip was the maximum the serial port could push per frame at 57600 baud. The firmware expands them to 38 LEDs using one of two modes, switchable from vvvv:

| Mode | Name | Behaviour |
|---|---|---|
| 0 | Double symmetry | Each strip driven independently. The 10 values are arranged in a double-mirror: the strip is split into two 19-LED halves, each half is a palindrome of the 10 values (v0…v9…v0). Both halves show the same thing. |
| 1 | All same, simple mirror | Both strips driven identically. All 20 buffer values (2×10) are treated as one sequence and mapped as a simple palindrome across the 38 LEDs. Strip independence is lost. |

Has a built-in standalone idle mode (slow hue cycling across both strips with a slight hue offset between them) that activates when no serial data is received for 10 seconds.

## V2.1 — 4 Strips, Arduino R3 (intermediate)

**Firmware:** `v2_ws2811/firmware/pitch_please_w2811_4-channel/pitch_please_w2811_4-channel.ino`

**Serial baud rate:** 57600

An intermediate step between v2.0 and v2.2. Four WS2811 strips were connected, but the Arduino R3 couldn't receive enough data per frame at 57600 baud to drive all four strips independently. The workaround was to receive full color data for only 2 strips and duplicate those colors to the other 2. This was a bandwidth limitation of the hardware.

The problem was solved by switching to the Arduino R4, which supports 921600 baud — high enough to send independent color data for all four strips per frame.

## V2.2 — 4 Strips, Arduino R4

**Firmware:** `v2_ws2811/firmware/pitch_please_w2811_4_channel_r4/pitch_please_w2811_4_channel_r4.ino`

**Serial baud rate:** 921600 (significantly higher than v2.0 — vvvv patch must use matching rate)

| Parameter | Value |
|---|---|
| Strips | 4 |
| LEDs per strip | 38 (physical) |
| Unique positions received | 19 (mirrored by firmware) |
| Data pins | D9, D10, D11, D12 |
| MCU | Arduino R4 |
| LED type | WS2811, FastLED |
| PSU | 12V, 6A |

**Protocol:** `(4 strips × 19 positions × 3 bytes RGB) + 2` bytes per frame.

The higher baud rate (921600 vs 57600) was needed to push data to 4 strips at adequate FPS. The vvvv patch was updated to match.
