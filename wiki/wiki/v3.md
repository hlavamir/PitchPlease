# Version 3 — ESP32 + DMX

> Current version: ESP32 receiving DMX512 input and driving 4× WS2811 LED strips. Two physical devices built, with hardcoded DMX start addresses 100 and 200.

## Overview

V3 is a major architecture shift. The microcontroller is now an ESP32 (AZ-Delivery USB-C variant) and the control protocol is DMX512 over RS-485, received via a MAX485 module on a custom PCB. The vvvv patch sends DMX from a computer via a USB-DMX interface (Enttec Pro or Enttec Open). Two identical devices have been built with different hardcoded DMX start addresses.

WiFi and Bluetooth are explicitly disabled in firmware to avoid interference with LED timing.

## Hardware

- **MCU:** AZDelivery ESP32 NodeMCU Module, CP2102, USB-C, ESP32-WROOM-32 ([Amazon](https://www.amazon.de/-/en/dp/B0D8WB3C22))
- **DMX connectors:** XLR-3 (3-pin)
- **DMX interface:** MAX485 RS-485 transceiver module
- **Level shifter:** POPESQ 74AHCT125N quad bus buffer (on PCB) — shifts ESP32's 3.3V data signals to 5V for the WS2811 strips (one IC, four channels)
- **LED strips:** 4× WS2811 12V, 24 segments per strip (72 LEDs per strip — 3 LEDs per WS2811 IC; 288 LEDs per device total). Adjacent strips are offset by 1/6 of a segment (≈8mm) so their LEDs interleave, reducing hotspots on the frosted tube.
- **PSU:** LXLQZXW 12V 10A 120W AC/DC adapter, 5.5×2.5/2.1mm plug ([Amazon](https://www.amazon.de/-/en/dp/B0C3L7MPY3))
- **PCB:** Custom design in Fritzing (`v3_esp32_dmx/hardware/250802_v3_esp32_dmx.fzz`), Gerber files exported August 2025 (`v3_esp32_dmx/hardware/250805 pcb/`)

## Physical Design

Each device is built around a **130 cm PLEXIGLAS® Satinice tube** (frosted/diffuse, colorless 0D010 DF, Ø60mm outer / Ø54mm inner). The four LED strips are glued to a **12×12mm hollow plastic square profile** (~1m, off-the-shelf from Baumarkt) running through the centre of the tube, one strip per face, so light is emitted in all four directions. The plastic profile has no structural role — it only keeps the LED strips in position. Structure is provided by the 3D printed caps and, in v3.1, the screw rod.

Each end of the tube gets a custom **3D printed cap**. The cap on one end houses all the electronics (PCB, connectors) and the XLR DMX connectors. The other end is a plain closing cap.

## DMX Channel Layout

Per device, starting at the device's DMX start address:

| Channels | Count | Purpose |
|---|---|---|
| +0 | 1 | Master dimmer |
| +1 | 1 | Mode |
| +2 … +5 | 4 | Per-strip dimmers (one per strip) |
| +6 … +77 | 72 | Group A: per-segment RGB (24 segments × 3 channels) |
| +78 … +80 | 3 | Group B |
| **Total** | **81** | channels per device |

## Connection to vvvv

The vvvv patch sends DMX to v3 devices via one of two paths:

- **Direct:** vvvv → Enttec Pro (USB-DMX) → XLR → device
- **Via ArtNet:** vvvv → local ArtNet → QLC+ → Enttec Open (USB-DMX) → XLR → device

Both devices can be addressed simultaneously since they have different start addresses (100 and 200).

## Deprecated Prototype

`standalone_esp32/` is an earlier prototype that explored a different architecture (audio analysis onboard the ESP32, web-based control UI over WiFi). It is deprecated and may be deleted. A future idea remains open: a WiFi bridge from a phone to vvvv (not directly to the ESP32) for remote light control — see Open Questions.

## Known PCB Issues

All three issues below require a PCB redesign — the board should not be fabricated again as-is. The next revision will be **v3.2**.

- **ESP32 solder holes too small** — the ESP32 footprint in the Fritzing part needs to be redesigned with larger through-holes.
- **MAX485 leg disconnection (DMX Out module)** — the RE and DE pins of the DMX Out MAX485 module were routed to GPIO33 on the ESP32. The intent was to drive them from firmware, but this couldn't be made to work. With GPIO33 undriven, DE/RE sit LOW — transmitter disabled, no DMX output. Workaround on assembled boards: physically cut the RE and DE legs of the module. PCB fix: remove the GPIO33 → RE/DE trace entirely. *(RE/DE legs identified from memory — verify against PCB.)*
- **Onboard step-down unreliable** — the PCB has a 3-pin footprint for a STPDWN step-down module: DollaTek ADS-5.5V, 4.75V–12V → 5V 800mA. This proved unreliable in practice (ESP32 only stayed powered for a few seconds — likely insufficient current headroom). Workaround: external DC-DC converter — YOURRYONG DE, DC 9V–36V → 5V 5A ([Amazon](https://www.amazon.de/-/en/dp/B0DSJR28JV)). PCB fix: remove the STPDWN footprint and route 5V from an external module instead.

## Future Development

### Axis flip mode
Rethink the way mode and per-channel dimmers work. Instead of the current DMX mode/per-strip-dimmer approach, add an **axis flip mode**: take the received 24-pixel brightness curve and resample it at 4 evenly-spaced positions (t = 0, 0.333, 0.666, 1.0 → approximately pixel indices 0, 8, 16, 23). Each of the 4 sampled brightness values drives one of the 4 LED strips as a uniform brightness. This maps the spatial distribution of the audio-reactive signal across strips, rather than addressing LEDs individually.

### WiFi bridge
Ability to control lights remotely from a phone. Not a direct phone-to-ESP32 connection — rather a bridge from the phone to vvvv, which then handles the DMX output as usual. See [[wifi-bridge]] for full details.
